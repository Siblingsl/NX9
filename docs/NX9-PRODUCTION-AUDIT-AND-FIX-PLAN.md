## 核心生产目标

优先保证这条主链路真实可跑通：

`剧本拆分 → 设定候选/设定检查 → 角色设定 + 场景设定 → 分镜网格 → 分镜预览（图像生成 + 3D 导演台）→ 批审 → 视频生成 → 拼接导出`

## 问题清单与解决方案

### P0-01：剧本拆分完成后自动写入角色/场景库，破坏“设定检查”门禁

现象：

- 剧本拆分节点完成后，`applyScriptBreakdownPayload()` 会自动把拆分出的角色和场景写入项目库。
- 这会导致设定检查节点在逻辑上变成“事后展示”，而不是“检查新角色/新场景并放行”。
- 用户在画布上很难判断哪些角色/场景是 AI 新推断出来的、哪些是项目原本已有的。

影响：

- 角色重复、场景重复、Prompt 漂移更难被发现。
- 设定检查节点的职责不清晰。
- 分镜网格预选框可能提前出现未经审核的资产。

解决方案：

- `applyScriptBreakdownPayload()` 默认只写入拆分结果和 storyboard，不自动写入角色/场景资产。
- 设定检查节点或候选面板显式执行“写入角色库/场景库”。
- 保留可选参数给必要场景使用，避免旧调用误触发自动入库。

状态：已修复。

修复记录：

- `applyScriptBreakdownPayload()` 增加 `syncAssets` 选项，默认不写入角色/场景库。
- 剧本拆分完成后只生成拆分结果、候选和 storyboard。
- 设定检查调用时显式传入 `syncAssets: true`。

### P0-02：角色/场景候选 Prompt 入口不统一，用户在画布节点上看不到

现象：

- 底部工作区有候选 Prompt，但画布上的剧本拆分节点原本没有直观入口。
- 用户以画布节点为主操作时，会误以为没有生成候选 Prompt。

影响：

- 剧本拆分 → 角色/场景设定的交接不透明。

解决方案：

- 画布节点增加“设定候选 / 查看 Prompt”入口。
- 候选项展示完整 Prompt、入库状态、复制 Prompt、写入库。

状态：已修复。

修复记录：

- 画布剧本拆分节点增加“设定候选 / 查看 Prompt”入口。
- 弹窗内展示完整角色/场景候选 Prompt、复制、写入库、入库状态。

### P1-01：候选 Prompt 生成逻辑在两个组件内重复

现象：

- 画布剧本拆分节点和底部工作区各自实现候选 Prompt 构造、入库、状态判断。

影响：

- 后续字段改动时容易一处修、一处漏。
- 角色/场景一致性规则难以维护。

解决方案：

- 抽出 `script-asset-candidates` 工具模块。
- 统一候选生成、Prompt 文本、已存在判断、场景库转换。

状态：已修复。

修复记录：

- 新增 `apps/web/src/engine/script-asset-candidates.ts`。
- 统一角色候选 Prompt、场景候选 Prompt、角色匹配键、场景候选转场景库、剪贴板复制兜底。
- 画布节点与底部工作区均改用统一工具。

### P1-02：设定检查对“场景”的匹配条件不完整

现象：

- 场景存在于 `environments` 与 `backlotWorkspace` 两套结构里。
- 检查时主要看 `environments`，而分镜预设又同时读取场景库。

影响：

- 可能出现“场景库已有，但设定检查仍显示新增”的误判。
- 或反过来：环境库有，但场景预选缺少。

解决方案：

- 设定检查同时检查环境库与场景库。
- 写入场景候选时同时写入 `environments` 和 `backlotWorkspace.scene`。

状态：已修复。

修复记录：

- 设定检查同时检查 `environments` 与 `backlotWorkspace.scene`。
- 设定检查同步新场景时，同时写入环境库与场景库。

### P1-03：Zustand selector 返回新数组会触发 React 无限刷新

现象：

- 之前出现 `getSnapshot should be cached` 和 `Maximum update depth exceeded`。
- 根因是 selector 内部直接 `filter()` 或 `?? []`。

影响：

- 打开节点后页面崩溃。

解决方案：

- selector 只取稳定对象/数组。
- 派生数组统一放入 `useMemo`。
- 后续新增 selector 需遵守这个规则。

状态：已修复已发现位置，复查通过。

复查记录：

- 对近期修改的剧本拆分、分镜网格、场景设定节点扫描，未发现 selector 内直接返回新数组的新增写法。

### P1-04：公共素材、私有素材、内置素材的“可编辑/可删除/可引用”边界需要更明确

现象：

- 公共内置素材应可引用但不可删除。
- 公共新增素材应可新增、编辑、删除。
- 私有素材应随项目保存。

影响：

- @ 引用、素材库新增、公共库管理容易出现 UI 行为不一致。

解决方案：

- 保持 `builtin` 只限制删除/覆盖，不限制引用。
- 新增按钮不应被内置素材隐藏。
- 公共库加载应在 @ 面板与素材库打开时都触发。

状态：已修复主要问题，待复查。

### P2-01：复制 Prompt 依赖 Clipboard API，失败时没有用户反馈

现象：

- `navigator.clipboard.writeText()` 在非安全上下文或权限被拒时可能失败。

影响：

- 用户点击复制后无感失败。

解决方案：

- 包装复制函数，失败时记录日志/提示用户手动复制。

状态：已修复。

修复记录：

- 候选 Prompt 复制改为 `copyTextWithLog()`，剪贴板失败时写入日志，提示用户手动复制。

### P2-02：部分网络型生成任务缺少超时、重试、继续轮询边界

现象：

- GrokGo/Magic Hour/Gateway 视频下载曾出现长时间生成、下载超时。

影响：

- 用户难以判断是仍在生成、下载失败还是网络失败。

解决方案：

- 网关层区分“上游生成中、下载失败、代理超时”。
- 前端展示可继续轮询/重试下载/使用远程 URL。

状态：待深入审计。

## 第二轮复查记录

### Gateway 视频生成/下载恢复性

发现：

- `proxyVideo()` 已经使用 `fetchWithTimeout()`，并能在下载远程视频失败时返回远程 URL，避免任务完全失败。
- `pollVideoTask()` 最多同步轮询约 90 秒；如果上游仍在生成，会返回 processing。
- 仍存在产品体验问题：前端需要明确区分“上游仍在生成”“本地下载失败但远程 URL 可用”“连接超时可重试”。否则用户会看到长时间“生成中”，不知道下一步该做什么。

建议下一轮修复：

- 前端视频节点保存 `taskId + provider/baseUrl + lastRemoteUrl + lastError`。
- 轮询超时后不显示失败，而显示“仍在上游队列，可稍后继续查询”。
- 本地下载失败但拿到远程 URL 时，UI 应显示“远程视频已生成，本地下载失败，可直接预览远程 URL / 重试下载”。
